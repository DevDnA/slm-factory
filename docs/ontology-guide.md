# 온톨로지 가이드

slm-factory의 온톨로지(지식 그래프) 기능을 활용하여 QA 생성 품질을 향상시키는 종합 가이드

---

## 1. 개요

### SLM 단독 QA 생성의 한계

slm-factory는 Teacher LLM이 문서를 읽고 QA 데이터를 생성합니다. 이 방식은 대부분의 경우 충분하지만, 다음과 같은 한계가 있습니다.

- **문서 간 관계 누락**: 각 문서를 개별적으로 처리하므로, 서로 다른 문서에 걸쳐 있는 개념의 연결을 파악하지 못합니다. 예를 들어, A 문서의 "보안 정책"과 B 문서의 "접근 권한 규정"이 관련된다는 사실을 인식하지 못합니다.
- **맥락 부족**: Teacher LLM이 문서 하나만 보고 질문을 생성하므로, 도메인의 큰 그림을 반영한 깊이 있는 질문이 나오기 어렵습니다.
- **중복과 편향**: 비슷한 내용의 문서에서 유사한 질문이 반복 생성될 수 있고, 관련 개념 간의 비교/대조 질문이 부족합니다.

### 온톨로지가 보완하는 방법

**온톨로지(Ontology)**는 문서에서 핵심 엔티티(사람, 조직, 개념, 기술 등)와 이들 간의 관계를 추출하여 **지식 그래프**를 구성합니다. 이 지식 그래프를 QA 생성 시 추가 맥락으로 제공하면:

- **관계 기반 질문 생성**: "A 기술과 B 기술의 차이점은?" 같은 관계 질문이 생성됩니다.
- **도메인 맥락 강화**: Teacher LLM이 엔티티 간의 관계를 인지한 상태로 질문을 생성하므로, 더 깊이 있는 QA가 만들어집니다.
- **중복 감소**: 이미 추출된 개념 체계를 참조하여, 다양한 관점의 질문이 생성됩니다.

> **비유**: 기존 QA 생성이 **문서를 한 장씩 읽는 것**이라면, 온톨로지를 활용한 QA 생성은 **마인드맵을 펼쳐놓고 문서를 읽는 것**과 같습니다. 개별 문서의 내용뿐만 아니라 전체 지식 체계 안에서의 위치를 인식하면서 질문을 만듭니다.

### slm-factory에서의 온톨로지

slm-factory의 온톨로지 기능은 **QA 생성 품질 향상**이라는 명확한 목적을 가집니다. 범용 지식 그래프 도구가 아닌, 파이프라인의 한 단계로서 설계되었습니다.

- **기본값: 비활성** — 단순한 문서 세트에서는 불필요할 수 있습니다.
- **선택적 활용** — `ontology.enabled: true`로 활성화, `ontology.enrich_qa: true`로 QA 생성에 반영
- **점진적 검증** — 먼저 추출 결과를 확인한 후, QA 반영 여부를 결정할 수 있습니다.

## 2. 온톨로지란?

### 지식 그래프 기본 개념

온톨로지(지식 그래프)는 정보를 **엔티티(Entity)**와 **관계(Relation)**로 구조화한 것입니다.

- **엔티티**: 문서에서 추출한 핵심 개체 (사람, 조직, 기술, 개념 등)
- **관계**: 엔티티 간의 연결 (예: "LoRA"는 "파인튜닝"의 한 방법이다)
- **트리플**: `(주어, 술어, 목적어)` 형식의 기본 단위 (예: `("slm-factory", "사용한다", "Teacher-Student 증류")`)

### slm-factory의 접근법

slm-factory는 **LLM 기반 추출 방식**을 사용합니다. 별도의 NER 모델이나 규칙 기반 시스템이 아니라, 이미 설정된 Teacher LLM이 문서에서 엔티티와 관계를 추출합니다.

- Teacher LLM이 문서를 읽고 엔티티/관계를 JSON으로 출력
- 신뢰도 점수(`confidence`)로 품질 필터링
- 문서 간 동일 엔티티 자동 병합 (대소문자 무시)
- 결과를 `ontology.json`에 저장

### slm-factory 온톨로지 vs 전문 지식 그래프 도구 비교

| 항목 | slm-factory 온톨로지 | 전문 KG 도구 (Neo4j 등) |
|------|----------------------|------------------------|
| **목적** | QA 생성 품질 향상 | 범용 지식 관리/쿼리 |
| **추출 방식** | Teacher LLM 기반 | NER + 규칙 + ML 파이프라인 |
| **엔티티 타입** | 7가지 (설정 가능) | 수백~수천 가지 |
| **쿼리** | 불가 (SPARQL 미지원) | 풍부한 그래프 쿼리 |
| **시각화** | 미지원 | 내장 또는 연동 |
| **추론** | 불가 | 규칙 기반 추론 가능 |
| **설치 비용** | 추가 설치 없음 | 별도 DB 서버 필요 |
| **사용 난이도** | YAML 설정 한 줄 | 스키마 설계 + 쿼리 학습 |

> slm-factory의 온톨로지는 "가벼운 지식 추출" 도구입니다. 전문적인 그래프 DB가 아니라, QA 생성이라는 단일 목적에 최적화된 보조 단계입니다.

## 3. 온톨로지 파이프라인

### 전체 흐름

온톨로지는 파이프라인에서 **Parse 이후, Generate 이전**에 실행됩니다.

```
문서 → Parse → [Ontology 추출] → Generate (+ 온톨로지 맥락) → Validate → ... → SLM
```

1. **Parse**: 문서를 텍스트로 변환
2. **Ontology 추출**: 각 문서에서 엔티티와 관계를 추출, 병합하여 `ontology.json` 저장
3. **Generate**: QA 생성 시 관련 온톨로지 맥락을 프롬프트에 추가 (`enrich_qa: true`인 경우)

### 추출 과정 상세

```
문서 텍스트 (content[:max_context])
    ↓ Teacher LLM
엔티티 추출 (이름, 타입, 신뢰도)
    ↓ min_confidence 필터링
관계 추출 (주어, 술어, 목적어, 신뢰도)
    ↓ min_confidence 필터링
엔티티 정규화 (대소문자 무시 병합)
    ↓
GraphStore에 병합
    ↓
ontology.json 저장
```

### 출력 형식

`ontology.json` 파일은 다음 구조로 저장됩니다:

```json
{
  "entities": [
    {
      "name": "LoRA",
      "entity_type": "Technology",
      "confidence": 0.95,
      "source_documents": ["파인튜닝_가이드.pdf"]
    }
  ],
  "relations": [
    {
      "subject": "LoRA",
      "predicate": "방법이다",
      "object": "파인튜닝",
      "confidence": 0.88,
      "source_document": "파인튜닝_가이드.pdf"
    }
  ]
}
```

## 4. 단계별 사용 가이드

다음 단계를 따라 온톨로지 기능을 활용할 수 있습니다. **3단계(검증)까지 먼저 진행한 후**, 결과가 만족스러우면 4단계에서 QA 생성에 반영하는 것을 권장합니다.

### Step 1: 온톨로지 활성화

`project.yaml`에서 온톨로지 설정을 활성화합니다.

```yaml
# project.yaml
ontology:
  enabled: true            # 온톨로지 추출 활성화
  entity_types:            # 추출할 엔티티 타입 (기본값 그대로 사용 가능)
    - Person
    - Organization
    - Concept
    - Technology
    - Document
    - Date
    - Location
  min_confidence: 0.5      # 최소 신뢰도 (낮을수록 더 많이 추출)
  max_concurrency: 4       # 동시 처리 문서 수
  enrich_qa: false         # ⚠️ 처음에는 false로 두고 결과 먼저 확인
  output_file: "ontology.json"
```

### Step 2: 온톨로지 추출 실행

`tool ontology` 명령으로 독립 실행하여 결과를 확인합니다.

```bash
# 온톨로지만 독립 실행
slm-factory tool ontology --config my-project/project.yaml
```

출력 예시:
```
✅ 온톨로지 추출 완료
   엔티티: 47개
   관계: 83개
   저장: my-project/output/ontology.json
```

### Step 3: 결과 검증

생성된 `ontology.json`을 열어 추출 품질을 확인합니다.

```bash
# JSON 파일 확인 (jq 사용 예)
cat my-project/output/ontology.json | python -m json.tool | head -50
```

**확인 포인트:**
- 엔티티가 도메인에 적절한가? (무관한 일반 단어가 많으면 `min_confidence` 올리기)
- 관계가 의미 있는가? (너무 자명한 관계만 있으면 QA에 기여도가 낮음)
- 누락된 핵심 개념이 있는가? (있으면 `entity_types`에 타입 추가 고려)

### Step 4: QA 생성에 반영

추출 결과가 만족스러우면 `enrich_qa`를 활성화합니다.

```yaml
# project.yaml
ontology:
  enabled: true
  enrich_qa: true    # ← QA 생성 시 온톨로지 맥락 사용
```

이후 `tool wizard` 또는 `run`을 실행하면, QA 생성 단계에서 자동으로 관련 온톨로지 맥락이 프롬프트에 포함됩니다.

```bash
# wizard로 전체 파이프라인 실행 (온톨로지 포함)
slm-factory tool wizard --config my-project/project.yaml
```

wizard에서는 **Step 3a (온톨로지 추출)**로 표시되며, Parse 후 자동으로 실행됩니다.

### Step 5: 품질 비교 (선택)

온톨로지 적용 전후의 QA 품질을 비교하려면:

```bash
# 1. enrich_qa: false로 QA 생성
slm-factory generate --config my-project/project.yaml
mv my-project/output/qa_pairs.json my-project/output/qa_without_ontology.json

# 2. enrich_qa: true로 QA 생성
# (project.yaml에서 enrich_qa: true로 변경 후)
slm-factory generate --config my-project/project.yaml
mv my-project/output/qa_pairs.json my-project/output/qa_with_ontology.json

# 3. 두 파일을 비교하여 질문 다양성, 관계 질문 비율 등 확인
```

## 5. 활용 시나리오

### 기술 문서 체계화

기술 표준, API 문서, 아키텍처 설계서 등 기술 문서가 서로 참조 관계를 가지는 경우. 온톨로지가 기술 간 의존성, 컴포넌트 간 관계를 파악하여 "A 기술이 B에 미치는 영향" 같은 연결 질문을 생성합니다.

- **적합한 문서**: API 레퍼런스, 시스템 설계서, 기술 표준 문서
- **기대 효과**: 기술 간 관계를 반영한 심층 QA 생성

### 조직 규정 연결

인사 규정, 보안 정책, 업무 매뉴얼 등 여러 규정이 상호 참조하는 환경. 온톨로지가 "이 규정은 저 규정의 예외 조항이다" 같은 규정 간 관계를 파악합니다.

- **적합한 문서**: 취업규칙, 보안 규정, SOP, 업무 매뉴얼
- **기대 효과**: 규정 간 관계와 예외 사항을 반영한 QA 생성

### 학술/연구 자료 정리

논문, 연구 보고서, 교재 등에서 연구자, 이론, 실험 결과 간의 관계를 추출. "A 연구자의 이론이 B 분야에 어떤 영향을 주었는가?" 같은 학술적 질문을 생성합니다.

- **적합한 문서**: 논문, 연구 보고서, 교과서, 기술 백서
- **기대 효과**: 연구 간 인용 관계, 이론 발전 과정을 반영한 QA 생성

## 6. 기술 조합 전략

### 7가지 조합 분석

SLM, RAG, 온톨로지는 각각 독립적으로 사용하거나 조합할 수 있습니다. **모든 조합이 항상 좋은 것은 아닙니다.** 상황에 맞는 조합을 선택하는 것이 중요합니다.

#### ① SLM 단독

| 항목 | 내용 |
|------|------|
| **장점** | 빠른 응답, 오프라인 동작, 프라이버시 보장, 낮은 운영 비용 |
| **단점** | 학습 시점 이후 정보 부재, 할루시네이션 가능, 출처 제시 불가 |
| **적합한 경우** | 변하지 않는 도메인 지식, 오프라인 환경, 단순 Q&A |
| **slm-factory 지원** | ✅ 핵심 파이프라인 (Parse → Generate → Train → Export) |

#### ② RAG 단독

| 항목 | 내용 |
|------|------|
| **장점** | 최신 정보 즉시 반영, 출처 명시 가능, 재학습 불필요 |
| **단점** | 검색 품질에 의존, 도메인 용어 이해 부족, 응답 지연 |
| **적합한 경우** | 자주 변경되는 문서, 출처 필수 요구, 프로토타입 구축 |
| **slm-factory 지원** | ❌ AutoRAG 등 별도 도구 사용 |

#### ③ 온톨로지 단독

| 항목 | 내용 |
|------|------|
| **장점** | 구조화된 지식 탐색, 관계 기반 탐색 가능, 지식 감사(audit) |
| **단점** | 자연어 질의응답 불가, 구축 비용, 지속적 유지보수 필요 |
| **적합한 경우** | 지식 체계 시각화, 데이터 감사, 관계 분석 목적 |
| **slm-factory 지원** | ✅ `tool ontology` 명령으로 독립 추출 가능 |

#### ④ SLM + RAG

| 항목 | 내용 |
|------|------|
| **장점** | 도메인 이해 + 최신 정보 + 출처 제시, 가장 널리 검증된 조합 |
| **단점** | 두 시스템 운영 복잡도, 검색 지연 추가 |
| **적합한 경우** | 대부분의 도메인 AI 시스템 (가장 범용적 조합) |
| **slm-factory 지원** | ✅ slm-factory + AutoRAG 연동 ([AutoRAG 연동 가이드](autorag-integration.html) 참조) |

#### ⑤ SLM + 온톨로지

| 항목 | 내용 |
|------|------|
| **장점** | 관계 인식 학습 데이터, 더 깊이 있는 QA, 체계적 지식 내재화 |
| **단점** | 온톨로지 품질이 학습 데이터에 직접 영향, 추출 비용 |
| **적합한 경우** | 개념 간 관계가 중요한 도메인 (법률, 의료, 기술 표준) |
| **slm-factory 지원** | ✅ `ontology.enrich_qa: true` 설정 |

#### ⑥ RAG + 온톨로지 (GraphRAG)

| 항목 | 내용 |
|------|------|
| **장점** | 구조화된 검색, 관계 기반 검색 가능, 더 정확한 문맥 제공 |
| **단점** | 구현 복잡도 높음, 운영 비용 2~3배, 전문 그래프 DB 필요 |
| **적합한 경우** | 대규모 지식 베이스, 복잡한 관계 질의가 빈번한 환경 |
| **slm-factory 지원** | ⚠️ 온톨로지 추출만 지원, GraphRAG 연동은 수동 통합 필요 |

#### ⑦ SLM + RAG + 온톨로지 (풀 스택)

| 항목 | 내용 |
|------|------|
| **장점** | 최고 수준의 도메인 AI — 지식 내재화 + 실시간 검색 + 구조적 이해 |
| **단점** | 최고 수준의 복잡도, 3개 시스템 운영/유지보수, 대부분의 경우 과잉 |
| **적합한 경우** | 대규모 엔터프라이즈 지식 관리, 복잡한 도메인 (금융 규제, 의료 프로토콜) |
| **slm-factory 지원** | ✅ slm-factory (SLM + 온톨로지) + AutoRAG (RAG)로 전체 스택 구성 가능 |

### 조합 선택 가이드

**어떤 조합을 선택해야 할까요?** 다음 질문에 답하며 결정하십시오.

1. **문서가 자주 변경되는가?** → 예: RAG 필요 (② ④ ⑥ ⑦)
2. **출처 제시가 필수인가?** → 예: RAG 필요 (② ④ ⑥ ⑦)
3. **도메인 전문 용어가 중요한가?** → 예: SLM 필요 (① ④ ⑤ ⑦)
4. **문서 간 관계가 복잡한가?** → 예: 온톨로지 고려 (③ ⑤ ⑥ ⑦)
5. **운영 복잡도를 감당할 수 있는가?** → 아니오: 단순 조합 선택 (① ② ④)

**권장 시작점:**
- **대부분의 경우**: ④ SLM + RAG로 시작
- **관계가 복잡한 도메인**: ⑤ SLM + 온톨로지로 시작, 필요시 RAG 추가
- **프로토타입**: ② RAG 단독으로 빠르게 검증 후 SLM 추가

### 주의사항

> **"온톨로지를 추가하면 무조건 좋아지는 것은 아닙니다."**

- **단순한 문서 세트**: FAQ, 단일 매뉴얼 등 문서 간 관계가 적은 경우, 온톨로지 추가는 비용만 늘리고 효과가 미미합니다.
- **Teacher LLM 품질 의존**: 온톨로지 추출 품질은 Teacher LLM의 능력에 직접 의존합니다. 작은 Teacher 모델로는 엔티티/관계 추출이 부정확할 수 있습니다.
- **노이즈 위험**: 품질이 낮은 온톨로지를 QA 생성에 반영하면, 오히려 QA 품질이 떨어질 수 있습니다. **반드시 Step 3(검증)을 거친 후** `enrich_qa`를 활성화하십시오.
- **추가 비용**: 온톨로지 추출은 Teacher LLM 호출이 추가되므로, 처리 시간과 API 비용이 증가합니다.

### slm-factory가 하는 것 / 하지 않는 것

| 하는 것 ✅ | 하지 않는 것 ❌ |
|-----------|---------------|
| LLM 기반 엔티티/관계 추출 | 외부 KG 연동 (Wikidata, DBpedia) |
| 신뢰도 기반 필터링 | SPARQL 쿼리 |
| 엔티티 정규화 (대소문자 무시 병합) | 그래프 시각화 |
| JSON 형식 저장 | 관계 추론 (삼단논법 등) |
| QA 생성 시 맥락 주입 | 그래프 알고리즘 (경로 탐색, 중심성) |
| 증분 업데이트 (변경/삭제 문서 반영) | 퍼지 매칭 (유사 이름 병합) |
| Wizard 통합 (Step 3a) | 스키마 강제 (관계 타입 제약) |
