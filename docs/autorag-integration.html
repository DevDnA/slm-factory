1. 개요

SLM 단독 사용의 한계

slm-factory로 만든 도메인 특화 SLM은 특정 분야에서 뛰어난 성능을 발휘하지만, 모든 상황을 완벽하게 대응하기는 어렵습니다. SLM 단독 사용 시 다음과 같은 한계가 존재합니다.

  • 할루시네이션(Hallucination): 학습 데이터에 포함되지 않은 질문에 대해 그럴듯하지만 부정확한 답변을 생성할 수 있습니다. 모델은 "모른다"고 답하는 대신 학습된 패턴을 조합하여 사실과 다른 내용을 만들어냅니다.
  • 최신 정보 부재: 파인튜닝 시점 이후에 추가되거나 변경된 정보를 반영할 수 없습니다. 규정 개정, 신규 장애 사례, 최신 기술 문서 등 지속적으로 갱신되는 정보에 취약합니다.
  • 근거 제시 불가: 답변의 출처를 명시할 수 없습니다. 사용자가 "어떤 문서의 몇 페이지에 근거한 답변인가?"라고 물으면 정확한 출처를 제공하지 못합니다.
  • 지식 범위 한정: 학습에 사용된 문서의 범위를 벗어나는 질문에는 효과적으로 대응할 수 없습니다.

RAG가 보완하는 방법

RAG(Retrieval-Augmented Generation)는 이러한 한계를 정확히 보완합니다. 사용자 질문이 들어오면 먼저 관련 문서를 검색하고, 검색된 문서를 컨텍스트로 제공하여 LLM이 답변을 생성하도록 합니다. 이 과정을 통해:

  • 할루시네이션 억제: 검색된 실제 문서에 근거하여 답변하므로, 근거 없는 내용 생성을 크게 줄입니다.
  • 최신 정보 반영: 벡터 DB에 새 문서를 추가하면 재학습 없이 즉시 최신 정보를 반영합니다.
  • 출처 명시: 답변에 사용된 원본 문서와 해당 구절을 함께 제공할 수 있습니다.
  • 지식 확장: 검색 범위를 넓히면 모델이 학습하지 않은 영역의 질문에도 대응할 수 있습니다.

slm-factory + AutoRAG 조합의 가치

slm-factory와 AutoRAG를 함께 사용하면 각각의 장점이 극대화됩니다.

비유: SLM은 전문가의 뇌이고, RAG는 참고 도서관입니다. 아무리 뛰어난 전문가라도 모든 세부 사항을 기억할 수는 없습니다. 하지만 잘 정리된 참고 도서관이 옆에 있다면, 전문가는 자신의 전문 지식으로 핵심을 파악하고 도서관에서 정확한 근거를 찾아 완벽한 답변을 제공할 수 있습니다. 범용 LLM에 RAG를 붙이는 것은 일반인에게 도서관을 제공하는 것이고, 도메인 특화 SLM에 RAG를 붙이는 것은 해당 분야 전문가에게 도서관을 제공하는 것과 같습니다.

slm-factory의 Teacher-Student 지식 증류를 통해 도메인 지식을 내재화한 SLM은 검색된 문서의 맥락을 더 정확하게 이해하고, 도메인 전문 용어를 적절하게 사용하며, 더 자연스럽고 정확한 답변을 생성합니다. AutoRAG는 이 SLM에 실시간 문서 검색 능력을 더하여, 할루시네이션 없는 도메인 전문 AI 시스템을 완성합니다.

2. AutoRAG란?

프로젝트 소개

AutoRAG는 RAG 파이프라인의 각 단계별 최적 전략 조합을 자동으로 탐색하는 오픈소스 프레임워크입니다. RAG 시스템을 구축할 때 가장 어려운 문제는 "어떤 검색 방식, 어떤 리랭킹 전략, 어떤 프롬프트 구성이 최적인가?"를 결정하는 것입니다. AutoRAG는 이 과정을 자동화합니다.

RAG 파이프라인 "조합"의 의미

RAG 파이프라인은 일반적으로 다음 단계로 구성됩니다. AutoRAG는 각 단계에서 여러 전략을 동시에 실험하고, 최적의 조합을 자동으로 찾아냅니다.

  • 검색(Retrieval): BM25, 벡터 검색, 하이브리드 검색 등 다양한 검색 전략
  • 리랭킹(Reranking): Cross-encoder, ColBERT, MonoT5 등 검색 결과 재순위화 전략
  • 프롬프트(Prompt): 검색 결과를 LLM에 전달하는 프롬프트 구성 전략
  • 생성(Generation): 응답을 생성하는 LLM 모델 및 파라미터 전략

AutoRAG는 이 모든 단계의 전략 조합을 자동으로 벤치마킹하여, 주어진 도메인 데이터에 최적화된 RAG 파이프라인 설정을 도출합니다.

slm-factory와 AutoRAG 비교

두 도구는 서로 다른 목적을 가지고 있으며, 함께 사용할 때 시너지를 발휘합니다.

항목 slm-factory AutoRAG
목적 도메인 특화 SLM 생성 최적 RAG 파이프라인 탐색
접근법 Teacher-Student 지식 증류 RAG 전략 벤치마크
입력 도메인 문서 도메인 문서 + QA 데이터셋
출력 파인튜닝된 SLM (Ollama) 최적화된 RAG 파이프라인 설정
문서 활용 QA 학습 데이터 생성 검색 코퍼스 (벡터 DB)

요약하면, slm-factory는 "모델 자체를 도메인에 맞게 만드는 도구"이고, AutoRAG는 "모델이 참조할 문서 검색 시스템을 최적화하는 도구"입니다. 두 도구를 결합하면 도메인 지식을 내재화한 모델이 최적화된 검색 시스템과 함께 동작하여 최고 수준의 도메인 AI를 구축할 수 있습니다.

3. 통합 아키텍처

전체 흐름

slm-factory와 AutoRAG의 통합은 하나의 도메인 문서 세트에서 시작하여 두 갈래 경로로 처리된 뒤, Ollama를 통해 하나의 통합 AI 서비스로 수렴합니다.

  • 왼쪽 경로 (slm-factory): 도메인 문서에서 QA 학습 데이터를 생성하고, Student 모델을 파인튜닝하여 Ollama에 도메인 특화 SLM을 배포합니다.
  • 오른쪽 경로 (AutoRAG): 동일한 도메인 문서를 벡터 DB에 색인하고, 최적의 RAG 파이프라인 구성을 자동으로 탐색합니다.
  • 수렴 지점 (Ollama): Ollama가 두 시스템의 다리 역할을 합니다. slm-factory의 export.ollama.model_name 설정으로 배포된 모델명이 AutoRAG YAML 설정의 model: 값과 동일하면, AutoRAG는 자동으로 slm-factory가 만든 SLM을 생성 모델로 사용합니다.

그림 1. slm-factory + AutoRAG 전체 통합 아키텍처 - Ollama가 두 시스템을 연결하는 다리 역할

데이터 흐름

동일한 도메인 문서가 두 가지 경로로 활용됩니다. slm-factory에서는 QA 학습 데이터로, AutoRAG에서는 검색 코퍼스(벡터 DB)로 사용됩니다. 사용자 질의가 들어오면 RAG가 관련 문서를 검색하고, 도메인 지식을 내재화한 SLM이 검색 결과를 기반으로 정확한 답변을 생성합니다.

그림 2. 데이터 흐름 - 빌드 타임(학습)과 런타임(질의) 시점의 상호 보완적 동작

4. 단계별 통합 가이드

다음 5단계를 순서대로 진행하면 slm-factory와 AutoRAG가 통합된 도메인 AI 시스템을 구축할 수 있습니다.

  1. slm-factory로 도메인 SLM 생성

    먼저 slm-factory를 사용하여 도메인 특화 SLM을 생성합니다. 프로젝트를 초기화하고, 도메인 문서를 추가한 뒤, wizard로 전체 파이프라인을 실행합니다.

    # 프로젝트 초기화
slm-factory init my-domain-project

# 도메인 문서 추가
cp /path/to/domain-docs/*.pdf my-domain-project/documents/
cp /path/to/domain-docs/*.hwpx my-domain-project/documents/

# wizard로 전체 파이프라인 실행
slm-factory tool wizard --config my-domain-project/project.yaml

    파이프라인이 완료되면 Ollama에 모델을 등록합니다. project.yaml의 export.ollama.model_name에 지정한 이름이 이후 AutoRAG에서 참조할 모델명이 됩니다.

    # project.yaml 설정 예시
export:
  ollama:
    model_name: "my-domain-model"
    system_prompt: "당신은 도메인 전문 AI 어시스턴트입니다."
    # 모델 배포
cd my-domain-project/output/merged_model
ollama create my-domain-model -f Modelfile

# 모델 동작 확인
ollama run my-domain-model "이 도메인에 대해 설명해주세요"
  2. AutoRAG 설치 및 데이터 준비

    AutoRAG를 설치하고, 벤치마크에 사용할 QA 데이터셋을 준비합니다. slm-factory에서 생성된 QA 데이터를 AutoRAG 형식으로 변환하여 활용할 수 있습니다.

    # AutoRAG 설치
pip install autorag

    AutoRAG에 필요한 데이터를 준비합니다. 코퍼스(검색 대상 문서)와 평가용 QA 데이터셋이 필요합니다.

    # 프로젝트 디렉토리 구성
mkdir autorag-project
mkdir autorag-project/corpus
mkdir autorag-project/data

# 도메인 문서를 코퍼스로 복사 (slm-factory와 동일한 문서 사용)
cp /path/to/domain-docs/*.pdf autorag-project/corpus/

    AutoRAG는 코퍼스를 자동으로 청킹하고 벡터 DB에 색인합니다. 평가용 QA 데이터셋은 slm-factory의 출력물(qa_alpaca.json)을 변환하여 사용하거나 직접 작성합니다.

  3. AutoRAG YAML에서 slm-factory 모델 연결

    AutoRAG의 YAML 설정 파일에서 생성(generator) 노드에 slm-factory가 배포한 Ollama 모델을 지정합니다. model 값은 slm-factory의 export.ollama.model_name과 반드시 동일해야 합니다.

    node_lines:
  - node_type: generator
    strategy:
      metrics: [bleu, rouge]
    nodes:
      - node_type: generator
        modules:
          - module_type: llama_index_llm
            llm: ollama
            model: my-domain-model
            batch: 1

    핵심 연결 포인트: slm-factory의 export.ollama.model_name: "my-domain-model" 값이 AutoRAG YAML의 model: my-domain-model과 일치해야 합니다. Ollama가 두 시스템 사이의 다리 역할을 하며, 이 모델명이 그 다리의 연결 고리입니다.

    범용 모델과 비교 벤치마킹도 가능합니다. 여러 생성 모듈을 추가하면 AutoRAG가 자동으로 비교합니다.

    node_lines:
  - node_type: generator
    strategy:
      metrics: [bleu, rouge]
    nodes:
      - node_type: generator
        modules:
          - module_type: llama_index_llm
            llm: ollama
            model: my-domain-model
            batch: 1
          - module_type: llama_index_llm
            llm: ollama
            model: qwen3:8b
            batch: 1
  4. AutoRAG 실행 및 최적 파이프라인 탐색

    AutoRAG를 실행하여 검색, 리랭킹, 프롬프트, 생성 단계의 최적 전략 조합을 자동으로 탐색합니다.

    # AutoRAG 최적화 실행
autorag evaluate \
  --config autorag-config.yaml \
  --corpus autorag-project/corpus \
  --qa autorag-project/data/qa.parquet \
  --project_dir autorag-project/results

    실행이 완료되면 각 단계별 최적 전략과 성능 메트릭을 확인할 수 있습니다. AutoRAG는 모든 조합을 벤치마킹한 후 최적의 파이프라인 설정을 자동으로 선택합니다.

    # 최적화 결과 확인
autorag dashboard --project_dir autorag-project/results
  5. 최적 RAG 시스템 배포

    탐색된 최적 파이프라인을 배포합니다. AutoRAG가 선택한 최적 설정을 사용하여 API 서버를 실행하면, slm-factory로 만든 도메인 SLM과 최적화된 RAG 파이프라인이 결합된 AI 서비스를 즉시 사용할 수 있습니다.

    # 최적 파이프라인으로 API 서버 실행
autorag deploy \
  --project_dir autorag-project/results \
  --host 0.0.0.0 \
  --port 8000
    # API 호출 테스트
curl -X POST http://localhost:8000/v1/run \
  -H "Content-Type: application/json" \
  -d '{"query": "이 규정의 예외 조항은 무엇입니까?"}'

    배포된 API는 사용자 질의를 받으면 다음 과정을 자동으로 수행합니다.

    1. 벡터 DB에서 관련 문서 검색 (AutoRAG 최적 검색 전략)
    2. 검색 결과 리랭킹 (AutoRAG 최적 리랭킹 전략)
    3. 검색된 문서를 컨텍스트로 구성 (AutoRAG 최적 프롬프트 전략)
    4. 도메인 SLM이 컨텍스트 기반 답변 생성 (slm-factory 모델, Ollama 경유)

5. 활용 시나리오

slm-factory + AutoRAG 조합은 도메인 문서가 지속적으로 축적되고, 정확한 근거 기반 답변이 필요한 모든 분야에 적용할 수 있습니다.

사내 규정 Q&A 시스템

인사 규정, 복리후생, 출장 정책 등 사내 규정 문서를 학습한 SLM과 최신 규정 문서를 검색하는 RAG를 결합합니다. 직원들의 규정 관련 질문에 정확한 조항과 출처를 제시하며 답변합니다.

SLM 역할: 규정 용어와 맥락을 이해하고 자연스러운 답변 생성

RAG 역할: 최신 규정 문서 검색, 해당 조항 번호와 원문 제시

적합한 문서: 취업규칙, 인사규정, 복무규정, 복리후생 안내, 출장여비규정

장애 대응 AI 어시스턴트

장애 매뉴얼과 과거 사고 보고서를 학습한 SLM이 장애 증상을 분석하고, RAG가 유사 장애 사례와 복구 절차서를 실시간으로 검색하여 대응 방안을 제시합니다.

SLM 역할: 장애 패턴 인식, 긴급도 판단, 복구 절차 요약

RAG 역할: 유사 장애 사례 검색, 최신 복구 절차서와 체크리스트 제공

적합한 문서: 장애 대응 매뉴얼, 사고 보고서(RCA), 복구 절차서, 시스템 아키텍처 문서

의료/법률 전문 자문 시스템

의료 가이드라인이나 법률 판례를 학습한 SLM이 전문 용어를 정확히 사용하고, RAG가 관련 조문이나 판례 원문을 검색하여 근거 있는 자문을 제공합니다. 전문 용어의 정확한 사용이 특히 중요한 분야입니다.

SLM 역할: 전문 용어 정확한 사용, 도메인 맥락 기반 판단

RAG 역할: 관련 조문/판례/가이드라인 원문 검색, 출처와 일자 명시

적합한 문서: 진료 가이드라인, 약물 정보, 법률 조문, 판례집, 행정 해석

6. 최적의 조합 전략

문서 분리 전략

slm-factory와 AutoRAG에 투입하는 문서의 역할이 다르므로, 문서 세트를 목적에 맞게 구성하는 전략이 필요합니다.

  • 공통 코퍼스 사용: 동일한 도메인 문서를 양쪽에 모두 활용하는 가장 간단한 방식입니다. slm-factory에서는 QA 학습 데이터로, AutoRAG에서는 검색 코퍼스로 사용합니다.
  • 핵심/보조 문서 분리: 핵심 도메인 지식(원칙, 절차, 기준)은 SLM 학습에 집중하고, 세부 참조 자료(사례, 부록, 첨부)는 RAG 검색 코퍼스에만 포함합니다. SLM은 핵심을 내재화하고, RAG는 세부 사항을 보완합니다.
  • 시간 기반 분리: 안정된 기존 문서는 SLM 학습에 사용하고, 지속적으로 갱신되는 최신 문서는 RAG 코퍼스에만 추가합니다. SLM 재학습 주기를 줄이면서 최신 정보를 즉시 반영할 수 있습니다.

모델 크기 선택

RAG와 함께 사용할 때 SLM의 크기 선택 기준이 달라집니다.

핵심 인사이트: RAG를 사용하면 SLM이 모든 지식을 기억할 필요가 없으므로, 1B~3B 규모의 소형 모델로도 충분한 성능을 달성할 수 있습니다. SLM의 역할은 "지식 저장"에서 "검색된 문서 이해 및 답변 생성"으로 전환됩니다.

모델 규모 SLM 단독 사용 SLM + RAG 결합
1B (예: gemma-3-1b-it) 제한적 도메인 이해, 단순 QA에 적합 RAG가 컨텍스트를 제공하므로 충분한 성능. 추론 속도 최적
3B (예: gemma-3-4b-it) 적정 수준 도메인 이해, 대부분 시나리오 대응 복잡한 질문 처리에 우수. 비용-성능 최적 균형
7B+ (예: qwen3:8b) 깊은 도메인 이해, 복잡한 추론 가능 성능은 뛰어나지만 추론 비용 증가. 고난도 도메인에 적합

일반적인 도메인 Q&A 시스템에서는 1B~3B 모델 + RAG 조합을 먼저 시도하고, 답변 품질이 부족할 때 모델 크기를 키우는 방식을 권장합니다.

업데이트 전략

도메인 지식이 변경되면 SLM과 RAG 양쪽 모두 업데이트가 필요합니다. slm-factory의 tool evolve 기능과 AutoRAG의 재최적화를 조합하여 지속적으로 시스템을 개선합니다.

# 1. 새 문서를 slm-factory에 추가하고 모델 진화
cp new-docs/*.pdf my-domain-project/documents/
slm-factory tool evolve --config my-domain-project/project.yaml

# 2. 동일한 새 문서를 AutoRAG 코퍼스에도 추가
cp new-docs/*.pdf autorag-project/corpus/

# 3. AutoRAG 재최적화 (새 코퍼스로 전략 재탐색)
autorag evaluate \
  --config autorag-config.yaml \
  --corpus autorag-project/corpus \
  --qa autorag-project/data/qa.parquet \
  --project_dir autorag-project/results

tool evolve는 증분 업데이트, 재학습, 품질 게이트, 버전 배포를 단일 명령으로 수행합니다. RAG 코퍼스는 새 문서를 추가하기만 하면 되므로, SLM 재학습보다 훨씬 빠르게 최신 정보를 반영할 수 있습니다.

성능 최적화 팁

  • 청킹 크기 조정: RAG의 문서 청킹 크기를 SLM의 max_seq_length에 맞추십시오. SLM이 처리할 수 있는 컨텍스트 범위 내에서 최대한 충분한 정보를 제공해야 합니다.
  • 리랭킹 활성화: 검색 결과의 리랭킹은 SLM이 관련성 높은 문서를 우선 참조하도록 하여 답변 품질을 크게 향상시킵니다. AutoRAG가 자동으로 최적 리랭킹 전략을 찾아줍니다.
  • 시스템 프롬프트 통일: slm-factory의 학습 시 사용한 시스템 프롬프트와 AutoRAG의 생성 시 사용하는 시스템 프롬프트를 일관되게 유지하십시오. 학습 시와 추론 시의 프롬프트 형식이 다르면 성능이 저하됩니다.
  • GGUF 양자화 활용: 배포 시 slm-factory의 tool gguf로 모델을 양자화하면 추론 속도가 향상됩니다. RAG의 검색 지연과 합산했을 때 전체 응답 시간을 최적화할 수 있습니다.
  • 평가 데이터 공유: slm-factory에서 생성된 QA 데이터를 AutoRAG의 벤치마크 데이터로 활용하면, 두 시스템이 동일한 기준으로 최적화됩니다.
